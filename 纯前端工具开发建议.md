> 写给：会提需求、会用 Claude Code、不会写代码的你

---

## 一、为什么纯前端最适合你

一个 HTML + CSS + JS 的单文件或几个文件的组合，具备以下特征：

| 特征 | 对你意味着什么 |
|------|--------------|
| 不需要后端/服务器 | 不用学数据库、不用买服务器、不用管运维 |
| 不需要用户系统 | 不用做注册登录、不用做密码找回、不用处理隐私合规 |
| 数据全在浏览器里 | 用户的数据存在他浏览器的 localStorage，你没有数据泄露的责任 |
| 部署就是上传文件 | GitHub Pages / Vercel 免费，拖上去就能访问 |
| 挂了刷新就好 | 不存在服务器宕机、数据库坏了这类灾难 |
| Claude Code 全能写 | 纯前端代码是 Claude 最擅长的领域，几乎不出错 |

**对你来说，前端项目的核心风险只有一个：代码逻辑写错了。而这个问题 Claude Code 自己就能修。**

## 二、选型三原则

在你不会写代码的前提下，挑项目要严卡三个条件：

### 原则 1：数据来源是自己脑子里有的，或者用户手动输入的

反例：想做「机票比价」→ 需要实时抓取各航司数据，纯前端做不到
正例：想做「家电维修决策器」→ 决策逻辑是自己编的，用户手动输入参数

### 原则 2：用户 30 秒内能理解这工具是干嘛的

反例：「一个基于语义理解的个人知识图谱构建器」→ 用户搞不懂
正例：「拍照/描述故障 → 告诉你修还是换」→ 小学生都懂

### 原则 3：能在一天内出一个能用的版本

反例：「一个完整的在线 PS 替代品」→ 几个月
正例：「订阅费用汇总计算器」→ Claude Code 两小时能写完

## 三、具体选题（按推荐度排序）

### 🥇 第一梯队：这些你现在就能做

#### 1. 「修还是换」家电决策器

**用到的数据**：你自己整理的维修价格区间 + 折旧规则（网上能搜到）
**输入**：家电类型、购买年份、购买价格、故障描述
**输出**：修还是换的建议 + 大概费用估算 + 推荐品牌（如果建议换）
**为什么好做**：核心就是 if-else 决策树，逻辑简单，非常适合 Claude Code 直接生成。做好了 SEO 长尾关键词（「冰箱不制冷值得修吗」）能吃搜索流量。

#### 2. 订阅管理面板（纯手动版）

**用到的数据**：用户自己输入
**输入**：订阅项目名称、金额、周期（月/年）、到期日、类别
**输出**：月度总支出、下月到期提醒、年度花费趋势图、分类占比饼图
**为什么好做**：CRUD + 图表，有大量现成 JS 库（Chart.js 等），Claude Code 直接写。数据全在用户浏览器 localStorage 里，你零隐私责任。

#### 3. 自由职业者收入平滑计算器

**用到的数据**：你自己设计的分配规则
**输入**：过去 12 个月每月收入
**输出**：建议每月给自己「发工资」多少、留多少做缓冲、建议应急储备金额
**为什么好做**：就是数学公式 + 简单的推荐逻辑，一天搞定。

#### 4. 「今天吃什么」决策器

**用到的数据**：你自己整理的中式家常菜数据库（50-100 道菜）
**输入**：冰箱里有哪些食材、几个人吃、想花多久做、忌口
**输出**：2-3 个推荐菜谱 + 步骤简述
**为什么好做**：标签匹配 + 随机推荐，逻辑极其简单但需求永远存在。

---

### 🥈 第二梯队：稍微复杂，但也能做

#### 5. 文件批量重命名预览器

**用到的数据**：不需要，纯逻辑
**功能**：用户选文件规则（替换/插入/删除/序号），实时预览重命名效果，然后一次性重命名
**为什么稍复杂**：需要用到 File System Access API，兼容性要考虑。但需求极强，每个人都遇到过。

#### 6. 文本对比工具（Diff Checker）

**用到的数据**：不需要
**功能**：粘贴两段文本 → 高亮显示差异 → 支持忽略空格/大小写
**为什么稍复杂**：Diff 算法本身不简单，但 Claude Code 能写出来。

#### 7. Markdown 编辑器 + 一键导出

**用到的数据**：不需要
**功能**：左边写 Markdown，右边实时预览，一键导出为 PDF（带中文支持）
**为什么稍复杂**：PDF 生成的中文字体处理是个问题。

### 🥉 第三梯队：有野心但慎重

#### 8. 个人财务仪表盘

手动录入收支记录 → 自动生成月度报告、分类折线图、预算预警。
**能做**，但因为涉及钱的敏感信息，用户会犹豫要不要信任一个没有品牌的单页工具。

#### 9. AI 聊天记录查看器（纯前端版）

类似 LLM Chat Explorer —— 拖入 AI 聊天导出 JSON，在浏览器里搜索和浏览。
**能做**，而且可以借鉴开源项目的逻辑。缺点是已经有竞品了。

## 四、技术选型：只用最省事的

建议让 Claude Code 使用以下技术栈（生成前直接告诉它）：

```
技术栈要求：
- 纯 HTML + CSS + 原生 JavaScript（不要 React/Vue 框架，杀鸡不用牛刀）
- 如果要图表，用 Chart.js CDN 引入
- 数据存储在 localStorage
- 所有样式写在一个 <style> 标签里
- 所有 JS 写在一个 <script> 标签里
- 不要任何 npm / build / 编译步骤
- 最终只需要一个 HTML 文件 + 可选的一个 CSS 文件就能跑
```

**为什么不用 React/Vue**：对你来说，框架增加的学习成本远超它带来的便利。单页工具几百行代码，原生 JS 完全够用。而且出问题时 Claude Code 能直接在单文件里定位修改，不用在组件树里绕来绕去。

## 五、怎么让 Claude Code 帮你写出来

### 第一步：写需求清单（你负责）

在跟 Claude Code 对话前，用文字把下面这些事想清楚：

```
1. 这个工具叫什么名字？（越直白越好，比如「修还是换」）
2. 用户打开页面第一眼看到什么？
3. 用户需要输入什么？（列出来每一个输入栏）
4. 输入完点一个按钮，会发生什么？
5. 输出结果长什么样？
6. 有没有「第二个功能」？（比如除了决策本身，还能导出结果）
```

例子（家电决策器）：

```
工具名：修还是换

打开页面 → 看到标题「这东西还值得修吗？」→ 一个表单
表单字段：
  - 家电类型（下拉：冰箱/空调/洗衣机/电视/热水器/其他）
  - 购买年份（数字输入）
  - 购买价格（数字输入，单位元）
  - 故障描述（多行文本，选填）
  - 一个绿色大按钮「帮我判断」

点击按钮 → 下面显示结果卡片：
  - 判断：建议维修 / 建议更换
  - 估算维修费用区间
  - 如果不修，推荐的新品价位和品牌
  - 决策理由（为什么这么判断）

第二个功能：页面底部可以点「再看一个例子」，随机加载一个预设案例展示判断过程
```

### 第二步：Claude Code 生成代码（Claude 负责）

把需求清单直接发给 Claude Code：

> 帮我写一个纯前端单页工具，[工具名]，需求如下：
> [粘贴你写的需求清单]
>
> 技术栈：纯 HTML + CSS + 原生 JavaScript，所有代码在一个 HTML 文件里。用 localStorage 存储用户数据。图表用 Chart.js CDN。

### 第三步：预览和调优（你试用，Claude 改）

Claude Code 生成 HTML 文件后，你直接双击在浏览器打开。然后：

1. 正常用一遍 → 看有没有哪里别扭
2. 故意乱输（年份写 2099、价格写负数）→ 看会不会崩溃
3. 手机上也打开试一下 → 看布局有没有炸
4. 把不爽的地方描述给 Claude Code → 让它改

这个循环重复 3-5 轮，基本上就成型了。

## 六、部署：让全世界能访问

### 方案 A：GitHub Pages（最推荐）

```
1. 在 GitHub 上新建一个仓库（用 Claude Code 帮你操作 gh 命令）
2. 把 HTML 文件推上去
3. 仓库 Settings → Pages → 选 main 分支 → Save
4. 几分钟后，你的工具就在 https://你的用户名.github.io/仓库名 上线了
```

优点：完全免费、HTTPS 自带、无限流量
缺点：域名是 github.io 的，不够酷

### 方案 B：Vercel

```
1. 把代码推到 GitHub
2. 去 vercel.com 注册 → 授权 GitHub → 导入仓库
3. 自动部署，给你一个 .vercel.app 域名
```

优点：比 GitHub Pages 快一丢丢
缺点：差不多，挑哪个都行

### 方案 C：买自己的域名

```
1. Namesilo / Namecheap 买域名（.com 大概 $10/年）
2. 在域名管理后台把域名指向 GitHub Pages 或 Vercel
3. 搞定
```

建议初期先用 github.io 域名验证，有人用了再买自己的域名。

## 七、上线之后做什么

### 1. 装统计
在你的 HTML 里让 Claude Code 加一段 Google Analytics 或 Umami（开源、隐私友好）统计代码。知道有没有人用、从哪来的。

### 2. 看看搜索流量
上线一周后去 Google 搜你工具能解决的问题（比如「冰箱坏了值得修吗」），看能不能搜到你的页面。搜不到就写一篇对应的文章链接过去。

### 3. 不做用户反馈系统
纯前端工具不要加反馈/留言功能——你没法自己维护后端和数据库。在页面底部放你的 GitHub 用户名或邮箱就够了。

### 4. 考虑开源
把代码公开在 GitHub 上。好处：
- 别人帮你看代码、提改进
- Google 给 GitHub 页面权重不错
- 万一有人想合作，会找上门

## 八、常见踩坑提醒

| 坑 | 怎么绕 |
|----|--------|
| **localStorage 只能存 5-10MB** | 单页工具够了。如果要存大量数据（比如聊天记录），用 IndexedDB |
| **不同浏览器数据不互通** | 告诉用户这个限制。或者做一个 JSON 文件导出→导入功能 |
| **清浏览器缓存数据就没了** | 加一个「导出数据 / 导入数据」按钮，让用户自己备份 JSON 文件 |
| **移动端布局爆炸** | 生成时务必提醒 Claude Code「移动端也要适配」 |
| **中文兼容性** | HTML 文件必须声明 `<meta charset="UTF-8">`，字体用系统默认中文字体 |
| **暗色模式** | 初期不做。纯前端工具，功能比好看重要一百倍 |
| **Chart.js CDN 加载失败（国内用户）** | 准备一个国内 CDN 替代链接（bootcdn 等），或者用纯 CSS 画简单的图 |
